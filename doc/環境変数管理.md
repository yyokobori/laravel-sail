# 環境変数管理とデグレード対策

## 概要

このプロジェクトは `conf/local/` ディレクトリの設定ファイルで運用環境ごとの設定を管理しています。
`start.sh` 実行時に `conf/local/` の設定が `app/backend/.env` に上書きされます。

**この機構を安全に保つため、新しい環境変数を追加する際は必ず `conf/local/.env.example` も更新してください。**

## ファイル構成

```
app/
  .env                    ← Docker Compose グローバル .env（自動生成）
  backend/
    .env                  ← Laravel 本体の環境変数（start.sh で生成）
    .env.example          ← Laravel 公式テンプレート（参考用）

conf/
  local/
    .env.example          ← 本番・開発共通の環境変数テンプレート（重要）
    mysql.cnf            ← MySQL 設定
    php.ini              ← PHP 設定
  aws/                    ← AWS環境用設定（将来追加予定）
```

## 【重要】新しい環境変数を追加する際の手順

### 1. `conf/local/.env.example` に追加

```bash
# conf/local/.env.example
MY_NEW_SETTING=value    # → コメントで何の設定かを説明
```

### 2. 変更内容をコミット

```bash
git add conf/local/.env.example
git commit -m "feat: 新機能 MY_NEW_SETTING を追加"
```

### 3. 開発チーム全員が `start.sh` を実行

```bash
./sh/start.sh
```

> `start.sh` は必須設定項目をバリデーションします。
> 見つからない項目がある場合は警告が表示されます。

## デグレード（機能喪失）を防ぐために

### ❌ してはいけないこと

1. **`conf/local/.env.example` を更新せずに環境変数を追加**
   - 他の開発者が `start.sh` を実行する時、新しい設定がなくなります
   - バグの温床になります

2. **`app/backend/.env` に直接編集**
   - 次回の `start.sh` 実行で上書きされます
   - `conf/local/.env.example` には反映されません

### ✅ すべきこと

1. **必ず `conf/local/.env.example` を更新**
   ```bash
   # 新機能を追加したら
   vim conf/local/.env.example
   ```

2. **コンテナを再起動して動作確認**
   ```bash
   ./sh/restart.sh
   ```

3. **必須設定項目のバリデーション結果を確認**
   ```
   ✓ 起動完了！
   ```

## よくある問題と解決方法

### 問題1: 新しい環境変数が反映されない

**原因** → `conf/local/.env.example` が更新されていない

**解決方法**
```bash
# conf/local/.env.example に必要な設定を追加
git diff conf/local/.env.example
./sh/start.sh
```

### 問題2: "必須設定項目が見つかりません" という警告が出た

**原因** → 新しく追加された環境変数が `conf/local/.env.example` に未登録

**解決方法**
```bash
# git の最新 conf/local/.env.example を確認
git status
git pull origin main

# または別のブランチから確認
git diff branch-name conf/local/.env.example

# 手動で追加
vim conf/local/.env.example
./sh/start.sh
```

### 問題3: デグレード（機能喪失）が疑われる

**チェックリスト**

```bash
# 1. conf/local/.env.example と app/backend/.env.example の内容を比較
diff conf/local/.env.example app/backend/.env.example

# 2. app/backend/.env と app/.env の内容を比較
diff app/backend/.env app/.env

# 3. 必須設定項目が揃っているか確認
grep "^DB_\|^APP_URL\|^VITE_" app/backend/.env

# 4. Docker Compose ログで環境変数が読まれているか確認
docker compose logs | grep "WARN\|ERROR"
```

## 設定項目の同期確認

### スクリプト: 設定項目を検証

```bash
#!/bin/bash
# 重要な設定項目をチェック（conf/local/.env.example が完全か確認）

LOCAL_ENV="conf/local/.env.example"
REQUIRED_KEYS=(
    "DB_HOST" "DB_PORT" "DB_DATABASE" "DB_USERNAME" "DB_PASSWORD"
    "APP_URL" "VITE_PORT" "VITE_API_BASE_URL"
    "LOG_CHANNEL" "LOG_LEVEL"
)

echo "=== 環境変数検証 ==="
for key in "${REQUIRED_KEYS[@]}"; do
    if grep -q "^${key}=" "$LOCAL_ENV"; then
        echo "✓ $key"
    else
        echo "✗ $key (見つかりません)"
    fi
done
```

## 定期メンテナンス

### 毎月の確認事項

1. **`conf/local/.env.example` が最新か確認**
   ```bash
   git log --oneline -n 10 -- conf/local/.env.example
   ```

2. **不要な設定項目がないか確認**
   ```bash
   grep "^#" conf/local/.env.example
   ```

3. **本番環境での設定値を確認**
   - 本番サーバーの `.env` が `conf/local/.env.example` より新しくないか確認

## まとめ

| 操作 | ファイル | 説明 |
|------|---------|------|
| 設定追加 | `conf/local/.env.example` | ここから始める（重要） |
| 確認 | `app/backend/.env` | `start.sh` で生成 |
| 確認 | `app/.env` | Docker Compose グローバル .env |
| 実行 | `./sh/start.sh` | 必須設定をバリデーション |

---

**デグレードを防ぐ鉄則：** 新しい環境変数が必要な場合は、必ず `conf/local/.env.example` を更新してコミットしてください。
