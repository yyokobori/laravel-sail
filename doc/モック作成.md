# モック運用整理（実装時 / 単体テスト）

## 概要

本資料は、フロント・バックエンド分離開発におけるモックの使い方を整理する。

---

# ■ 1. 実装時のモック

## 目的

- API未完成でも画面開発を進める
- バックエンドの疎通確認を早期に行う
- データ構造（OpenAPI準拠）を先に固定する

---

## フロント側

### 使用方法

- Swagger（OpenAPI）を使用する
- API仕様を元にモックレスポンスを生成する

---

### 確認内容

- レスポンス構造
- フィールド名
- enum（code / name）
- null許可 / 必須項目

---

## バックエンド側

### Service

- モックレスポンスを返却する
- Resource形式に完全一致させる

---

### Request

- バリデーション定義の確認
- 型・必須・形式が正しいか

---

### 確認内容

- APIレスポンス形式がOpenAPIと一致しているか
- フロントがそのまま使えるか
- バリデーションが正しく動作するか

---

# ■ 2. 単体テスト時のモック

## 目的

- Serviceロジックの正当性検証
- 依存（DB / 外部API）の排除
- 回帰防止

---

## フロント側

### 使用方法

- Swagger（OpenAPI）とレスポンスを照合する

---

### 確認内容

- APIレスポンスが仕様と一致しているか
- フィールド構造が変わっていないか

---

## バックエンド側

### Service

- Repositoryをモック化する
- Service単体で動作確認する

---

### Request

- バリデーションのテストを行う
- 正常 / 異常ケースを検証する

---

### 確認内容

- ロジックが正しいか
- エラー処理が正しいか
- バリデーションが期待通りか

---
# ■ Swagger（OpenAPI）による確認方法（ざっくり）

---

## 環境

- Swagger UI: `http://localhost:8080`
- OpenAPI YAML: `/app/app/openapi/*.yaml`

---

## 使用方法

### 手動リクエスト

- フロントからSwagger UI記載のmocのURLをたたく
- リクエスト内容・パラメータを確認
- レスポンス構造、型、enum、null許可などを確認

---

### レスポンス調整

- 値を変えて確認したい場合は OpenAPI YAML の `response` 定義を編集
- これによりフロント開発時に任意のレスポンスを確認可能

---

### ■ モックサーバ（必要があれば）

- OpenAPIからモックサーバ生成（Prismなど）
- フロント単独で開発・動作確認可能
- レスポンスの値を自由に変えてテスト可能

---

## 参考リンク（日本語）

# ■ Laravel Mockery モックテスト（ざっくり）
---

- Service 層を Interface モック で置き換え
- Request の Validator も確認している
- PHPUnit + Mockery で完結

```php
<?php

namespace Tests\Unit\Services;

use Tests\TestCase;
use Mockery;
use App\Services\UserServiceInterface;
use App\Http\Requests\UserCreateRequest;
use Illuminate\Support\Facades\Validator;

class UserServiceTest extends TestCase
{
    protected function tearDown(): void
    {
        Mockery::close(); // Mockeryのクリーンアップ
        parent::tearDown();
    }

    /** @test */
    public function create_user_calls_service_correctly()
    {
        // 1. モック作成
        $userServiceMock = Mockery::mock(UserServiceInterface::class);
        $userServiceMock
            ->shouldReceive('create')
            ->once()
            ->with(['name' => 'test', 'email' => 'test@example.com'])
            ->andReturn(['id' => 1, 'name' => 'test', 'email' => 'test@example.com']);

        // 2. DI でモックをバインド
        $this->app->instance(UserServiceInterface::class, $userServiceMock);

        // 3. 実際に呼び出す
        $request = new UserCreateRequest([
            'name' => 'test',
            'email' => 'test@example.com',
        ]);

        $response = app(UserServiceInterface::class)->create($request->all());

        // 4. 結果を検証
        $this->assertEquals('test', $response['name']);
        $this->assertEquals(1, $response['id']);
    }

    /** @test */
    public function request_validator_passes()
    {
        $data = ['name' => 'test', 'email' => 'test@example.com'];
        $request = new UserCreateRequest($data);

        $validator = Validator::make($request->all(), $request->rules());
        $this->assertFalse($validator->fails());
    }
}
```
