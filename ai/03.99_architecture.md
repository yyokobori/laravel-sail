# React Architecture（03.99）

更新日: 2026-03-06

## 目的
- 実装の責務境界を明確にし、デグレを防止する。
- 参照先を一元化し、AI実装時のブレを抑える。

### 各レイヤーの責務
- `view/`
  - 画面単位の構成と表示制御を担当する。
  - 区分は `public / user / admin / errors` とする。
  - 直接API通信は `api/` または `hooks/` を経由する。
- `component/`
  - 再利用UI部品を担当する。
  - `component/***/` 形式で配置し、`component/` 直下にファイルを置かない。
  - 各ディレクトリの公開窓口は `index.ts` に統一する。
  - 詳細は `ai/03.03_component.md` を参照する。
- `template/`
  - `public / user / admin` の共通レイアウト枠を担当する。
  - `reset.css` と `tailwind.css` の読み込み起点とする。
- `api/`
  - バックエンド通信処理を担当する。
  - API呼び出し実体はこの配下に集約する。
- `types/`
  - 共通型・API関連型・画面利用型を管理する。
- `hooks/`
  - 再利用可能な状態管理・副作用処理を担当する。
  - 命名は `useXxx` に統一する。

## 依存方向ルール（確定）

### 許可依存
- `view` → `hooks`, `api`, `component`, `template`, `types`
- `hooks` → `api`, `types`
- `component` → `types`
- `template` → `component`, `types`（必要時のみ）
- `api` → `types`
- `types` → （依存なし）

### 禁止依存
- `component` → `api`（直接通信禁止）
- `component` → `view`
- `api` → `view` / `component` / `template`
- `types` → 他レイヤ
- `view` → 他 `view` への直接依存

### 運用ルール
- 共通化が必要な場合は `component` / `hooks` / `types` へ切り出す。
- 例外が必要な場合は、理由・影響範囲・代替不可理由を仕様に記載してから実装する。
- 変更時は依存違反がないかをレビューで確認する。

## 状態管理方針（確定）
- ローカル状態は `useState` を基本とする。
- 複雑な状態遷移は `useReducer` を使用する。
- 外部ストア（Zustand等）は初期導入しない。
  - 必要性が明確になった時点で導入判断する。

### `useState` / `useReducer` の使い分け
- `useState` を使う条件
  - 状態が単純で、更新パターンが少ない。
  - 1画面または1コンポーネント内で閉じる。
- `useReducer` を使う条件
  - 状態項目が多く、更新ルールが相互依存する。
  - イベント起点の状態遷移を明確に管理したい。

### 外部ストア導入の判断条件
- 同じ状態を3画面以上で共有する。
- props バケツリレーが3階層以上で継続的に発生する。
- 複数レイアウトで同一状態を頻繁に参照する。

### 注意事項
- `component/**` に業務状態を持ち込みすぎない。
- APIレスポンスとUI状態を混在させない。

## データ取得方針（確定）
- API通信は `api/**` に集約する。
- `view/**` と `component/**` で直接 `fetch` しない。
- データ取得の起点は `view/**` または `hooks/**` に限定する。

## エラー/バリデーション方針（確定）
- バリデーションはすべてバックエンドで実施する。
  - フォーム送信前チェックもバックエンドへ送信して判定する。
- エラー表示は、テンプレート利用先アプリ側の仕様（デザイン）に準拠する。

## 変更時チェック（最低限）
- 既存呼び出しへの影響有無を確認する。
- 型チェックとビルドを通過させる。
