# 02.08 Errorルール

## ■ 目的

APIのエラー処理を統一し、  
フロントエンド連携および障害調査を容易にする。

---

## ■ 責務

- エラーレスポンスの統一
- 例外処理の一元管理
- ログとの連携

---

## ■ エラー処理方針

- エラーは原則として例外（throw）で処理する
- エラー処理はグローバルハンドラで一元管理する
- Controllerで個別にエラーハンドリングを行わない

---

## ■ エラーレスポンス

- HTTPステータスコードでエラー種別を表現する
- フロントには汎用メッセージのみ返却する
- 詳細なエラー情報は返却しない

例：

```json
{
  "error": {
    "message": "エラーが発生しました"
  }
}
```

## ■ ステータスコード

---
| 種別         | コード |
| ---------- | --- |
| バリデーションエラー | 400 |
| 認証エラー      | 401 |
| 権限エラー      | 403 |
| Not Found  | 404 |
| サーバーエラー    | 500 |
---

## ■ バリデーションエラー（例外）

バリデーションエラーのみ例外として扱う

FormRequestが項目単位のエラーを返却する

例：
```json
{
  "errors": {
    "email": ["メールアドレスが不正です"]
  }
}
```

## エラーコード（内部管理用）
エラーコードは内部管理用として定義する
APIレスポンスには出力しない
定義場所：app/Constants/Errors.php

## ■ フォーマット
```code
E + HTTPステータス + 000
```

例：
- E400000
- E401000
- E403000
- E404000
- E500000

## ■ メッセージ管理

エラーメッセージは一元管理する

resources/lang/言語 配下で管理する

コード上で直接記述しない

## ■ 例外ハンドリング

例外は app/Exceptions/Handler.php で処理する

想定外例外はすべて500として扱う

スタックトレースはレスポンスに含めない

## ■ ログ連携

errorレベル以上は必ずログ出力する

request_id を付与する

ログとエラーを紐付け可能にする

## ■ 禁止事項

Controllerでの個別エラーハンドリング

エラー内容の詳細返却

生の例外メッセージ返却

try-catchの乱用

スタックトレースの返却

エラーコードのレスポンス出力

## ■ 実装方針

エラーは必ず例外で処理する

レスポンスはシンプルに保つ

ログで詳細を管理する

## ■ ゴール

エラー処理が全APIで統一されること

フロント側で一貫したエラーハンドリングが可能になること

障害調査が容易になること
