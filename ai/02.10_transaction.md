# 02.10_transaction.md

## 概要

本仕様は、Laravelバックエンドにおけるトランザクション管理ルールを定義する。

目的：

- データ整合性の担保
- 複数処理の原子性確保
- エラー発生時のロールバック統一
- AI実装時のトランザクション漏れ防止

---

## 基本方針（最重要）

- トランザクションはService層でのみ管理する
- Controller / Repository でのトランザクションは禁止
- 1業務処理 = 1トランザクションとする
- 例外発生時は必ずロールバックする
- commit/rollbackは自動化する

---

## トランザクション責務

| レイヤー | トランザクション |
|----------|----------------|
| Controller | 禁止 |
| Request | 禁止 |
| Service | 実行 |
| Repository | 禁止 |
| Model | 禁止 |

---

## 実装方針

- BaseServiceでトランザクション処理を共通化する
- DB::transaction または beginTransaction を使用する
- 例外はそのままthrowする

---

## 基本実装（推奨）

BaseServiceに共通処理を定義する：

abstract class BaseService
{
    protected function executeInTransaction(callable $callback)
    {
        return \DB::transaction(function () use ($callback) {
            return $callback();
        });
    }
}

---

## 使用例

public function createUser(array $data)
{
    return $this->executeInTransaction(function () use ($data) {

        $user = $this->userRepository->create($data);

        return $user;
    });
}

---

## 例外との関係（重要）

- 例外がthrowされた場合、自動でrollbackされる
- Serviceでcatchしない
- Handlerに処理を委譲する

NG例：

try {
    // 処理
} catch (\Exception $e) {
    // ログだけ出して終了 ← 禁止
}

---

## Loggingとの連携

トランザクション処理はログと連動する

- 開始ログ（任意）
- 成功時ログ（info）
- 失敗時ログ（error）

例：

$this->executeInTransaction(function () {
    // 処理
});

---

## パフォーマンスログとの連携

- トランザクション単位でdurationを計測する
- BaseServiceのパフォーマンスログと統合する

---

## ネストトランザクション

### 方針

- 原則禁止
- ServiceからServiceを呼ばない設計とする

---

### 理由

- 意図しないcommit/rollbackが発生する
- 可読性が低下する
- AI実装でバグの原因になる

---

## トランザクション分離レベル

- デフォルト設定を使用する（READ COMMITTED）
- 特別な要件がない限り変更しない

---

## 外部APIとの関係

- 外部API呼び出しはトランザクション外で実行する

理由：

- ロールバックできないため
- タイムアウトリスクがあるため

---

## 禁止事項

- Repositoryでトランザクションを開始する
- Controllerでトランザクションを管理する
- try-catchで握り潰す
- 外部APIをトランザクション内で実行する
- ネストトランザクションを使用する

---

## AI向け実装指針

- DB更新が複数ある場合は必ずトランザクションを使用する
- BaseServiceのメソッドを利用する
- 例外は必ずthrowする
- commit/rollbackは自動に任せる
- Service外でトランザクションを書かない
