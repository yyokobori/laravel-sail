# 02.01 アーキテクチャルール

## ■ 目的

Laravelバックエンドにおける責務分離と依存関係を定義し、
実装の一貫性と保守性を担保する。

---

## ■ レイヤー構造

本プロジェクトでは、以下の構造を厳守する：

Controller → Service → Repository → Model

---

## ■ データフロー

リクエストからレスポンスまでの流れ：

Request → Controller → Service → Repository → Model → DB
DB → Model → Repository → Service → Resource → Response

---

## ■ 依存関係ルール

* ControllerはServiceにのみ依存する
* ServiceはRepositoryにのみ依存する
* RepositoryはModelにのみ依存する
* Modelは他レイヤーに依存しない

逆方向の依存は禁止する

---

## ■ 各層の責務

### Controller

* HTTPリクエストを受け取る
* FormRequestでバリデーションを行う
* Serviceを呼び出す
* Resourceでレスポンスを返す

### Service

* ビジネスロジックを実装する
* トランザクション制御を行う
* 複数Repositoryを組み合わせる

### Repository

* データベース操作を担当する
* クエリの集約
* Modelの操作をラップする

### Model

* テーブル定義
* リレーション定義
* 属性定義

---

## ■ 実装フロー（必須）

AIは以下の順序で実装すること：

1. Request作成（バリデーション定義）
2. Service作成（ロジック定義）
3. Repository作成（DB操作）
4. Controller作成（入口処理）
5. Resource作成（レスポンス整形）

---

## ■ トランザクションルール

* トランザクションはService層でのみ使用する
* Repositoryではトランザクションを開始しない

---

## ■ 禁止事項

* ControllerからRepositoryを直接呼び出す
* ServiceからModelを直接操作する
* Repositoryを経由せずDB操作を行う
* Modelにビジネスロジックを書く
* レイヤーをまたいだ責務の混在

---

## ■ 例外的ケース

例外的に構造を変更する場合：

* 必ず理由をコメントとして明記する
* 既存構造を壊さない範囲で行う

---

## ■ ゴール

* すべての処理が明確なレイヤーに配置される
* 依存関係が一方向に保たれる
* 修正時の影響範囲が限定される
* AIが構造を迷わず実装できる
