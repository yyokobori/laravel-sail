# Laravelログ運用仕様

更新日: 2026-03-10

## 目的
- Laravel backend のログ運用を統一し、障害調査性と保守性を高める。
- 機密情報の漏えいを防ぎ、安全に運用する。

## ログレベル定義

| レベル | 用途 | 本番出力 | 例 |
|--------|------|----------|-----|
| debug | デバッグ時の詳細情報 | なし | 変数の中間値、SQL詳細、実装中の挙動確認 |
| info | 通常処理の記録 | あり | ログイン成功、バッチ開始/終了、業務イベント完了、単発の401/403/404 |
| warning | 注意が必要な状態 | あり（監視対象） | 外部API遅延、リトライ発生、連続401/403/404、復旧可能な異常 |
| error | 例外発生時 | あり（通知対象） | 未処理例外、DB接続エラー、5xxエラー、業務処理失敗 |
| critical | 重大障害 | あり（即時通知） | システム全体障害、データ整合性問題、重要連携停止 |

## ログ出力ドメイン仕様

### 目的
- 障害の検知と調査を分離し、運用時の初動と原因追跡を容易にする。
- 環境ごとの必要最小限のログ出力により、可観測性と保守性を両立する。

### ログファイル定義
| ファイル | 用途 | 出力レベル |
|---|---|---|
| develop.log | 開発時の詳細確認 | debug, info, warning, error, critical |
| error.log | 障害検知・通知 | warning, error, critical |
| info.log | 障害調査・業務追跡 | info, warning, error, critical |

### 環境別出力方針
| 環境 | 出力ファイル |
|---|---|
| local | develop.log, error.log, info.log |
| stg | error.log, info.log |
| prd | error.log, info.log |

### ローテーション
- すべてのログファイルを日次ローテーションする。
- 保持日数は環境別に設定する（例: local 7日 / stg 14日 / prd 30日）。

## ログ出力フォーマット仕様

- すべてのログは下記の形式で出力する。
- サンプル（infoレベルの場合）:

```
[info] ---
ログの内容（省略）
-------------------
```

- 各レベル（debug, info, warning, error, critical）で [level] 部分が変化する。
- 本フォーマットは Monolog のカスタムフォーマッター（CustomLineFormatter）で実現する。
- フォーマッターの実装例:

```php
// app/Logging/Formatters/CustomLineFormatter.php
class CustomLineFormatter extends LineFormatter
{
    public function __construct()
    {
        $format = "[%level_name%] ---%extra.line_break%%message%%extra.line_break%-------------------%extra.line_break%";
        parent::__construct($format, null, true, true);
    }
    public function format(array $record): string
    {
        $record['extra']['line_break'] = "\n";
        return parent::format($record);
    }
}
```

- config/logging.php の各チャネルで `formatter` にこのクラスを指定することで適用できる。

## 実装方法

### .env 設定
保持日数は `.env` で環境別に定義する：

```
LOG_DEVELOP_DAYS=7
LOG_ERROR_DAYS=14
LOG_INFO_DAYS=14
```

### config/logging.php 設定
Laravel の `daily` ドライバーを使用し、3つのチャネルを定義する：

```php
'channels' => [
    'develop' => [
        'driver' => 'daily',
        'path' => storage_path('logs/develop.log'),
        'level' => 'debug',
        'days' => env('LOG_DEVELOP_DAYS', 7),
    ],
    'error' => [
        'driver' => 'daily',
        'path' => storage_path('logs/error.log'),
        'level' => 'warning',
        'days' => env('LOG_ERROR_DAYS', 14),
    ],
    'info' => [
        'driver' => 'daily',
        'path' => storage_path('logs/info.log'),
        'level' => 'info',
        'days' => env('LOG_INFO_DAYS', 14),
    ],
],
```

### アプリケーションでの使用
環境に応じてデフォルトチャネルを切り替える：

```php
// local環境
LOG_CHANNEL=develop

// stg/prd環境
LOG_CHANNEL=stack
LOG_STACK=error,info
```

## 実装クラス・自動付与項目・マスキング仕様

### 主な実装クラスの責務
- `App\Logging\LogContext`：リクエスト単位で request_id, user_id, route など共通情報を保持
- `App\Logging\Processors\ContextProcessor`：全ログに共通情報（request_id等）を自動付与
- `App\Logging\Processors\SensitiveDataMaskingProcessor`：パスワード・トークン・メール等の機密情報を自動マスキング
- `App\Http\Middleware\AssignRequestId`：リクエストごとに一意なIDを生成し、LogContextにセット＆レスポンスヘッダーに付与
- `App\Providers\LoggingServiceProvider`：上記の初期化・全チャネルへのプロセッサ登録・環境別デフォルト切替

### ログに自動付与される項目
- request_id（リクエスト単位の一意ID）
- user_id（認証済みの場合のみ）
- route（ルート名）
- http_method（GET/POST等）
- url（リクエストURL）

### マスキング対象例
- "password"、'password'、"token"、Bearerトークン
- メールアドレス、クレジットカード番号
- これらは SensitiveDataMaskingProcessor で正規表現により自動置換

### カスタムフォーマッター適用例
- config/logging.php の各チャネルで `formatter` に CustomLineFormatter::class を指定
- 例：
```php
'info' => [
    'driver' => 'daily',
    'path' => storage_path('logs/info.log'),
    'level' => 'info',
    'days' => env('LOG_INFO_DAYS', 14),
    'formatter' => App\Logging\Formatters\CustomLineFormatter::class,
],
```
