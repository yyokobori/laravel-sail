# 認証フロー仕様

更新日: 2026-03-06

## 前提
- 本仕様では、`laravel(backend)` を認証・認可の最終判定主体とする。
- roleは `ai/04_role.md` の定義（`User` / `Admin`）に従う。

## 用語
- 未認証: セッション未確立、または有効期限切れ。
- 認証済み: セッションが有効で、ユーザー識別が可能な状態。
- 権限不足: 認証済みだが要求リソースへのアクセス権がない状態。
- セッションタイムアウト: 認証済み操作中にセッションが失効し、`401` が返る状態。
- 本仕様のエラー種別表記はHTTPステータスコード（例: `404`）で統一し、`not-found` はURL/パス名としてのみ使用する。

## laravel(backend) 側の責務
- ログイン成功時にセッションを確立する。
- 未認証アクセスには `401` を返す。
- 権限不足アクセスには `403` を返す。
- 認証状態確認API（例: current-user）で現在ユーザー情報を返す。

## react(frontend) 側の責務
- ルーティング層で `public / user / admin` の境界を判定する。
- 画面初期化時に認証状態確認APIを呼び出し、判定結果で遷移制御する。
- `401` 受信時はログイン導線へ遷移する。
- `403` 受信時は権限エラー画面へ遷移する。

## ルートガード方針
- `public` ルートは常時アクセス可能。
- `user` ルートは認証済みかつ `User` または `Admin` を許可する。
- `admin` ルートは認証済みかつ `Admin` のみ許可する。
- ガード判定はUI上の事前制御であり、最終判定は `laravel(backend)` が行う。

## ルートガード実装ルール
- 判定順序は `認証状態 -> role -> 遷移先決定` に固定する。
- エラー競合時の優先順位は `未認証(401) -> 権限不足(403) -> 404` とする。
- リダイレクト前に「現在URLと遷移先URLが同一なら遷移しない」判定を必須とする。
- `login` / `forbidden` 画面では再ガードによる循環遷移を発生させない。
- ガード結果とAPI結果が不一致の場合は、API結果（`401/403`）を優先する。

## APIエラー時の遷移方針
- `401`（未認証 / session-timeout）: ログイン画面へ遷移（必要に応じて復帰先URLを保持）。
- `403`: `view/errors/forbidden` を表示。
- `404`: `view/errors/not-found` を表示。
- `5xx`: `view/errors/server-error` を表示。
- ネットワーク断など通信失敗: アプリ仕様に準拠して再試行導線または汎用エラー表示。

## 復帰先URLルール
- 保持は `401`（未認証 / session-timeout）で `public` 以外にアクセスした場合のみ行う。
- 保持対象は内部URLの `path + query` を基本とし、危険な再実行を伴う操作起点は保持しない。
- `login` / `forbidden` / `not-found` / `server-error` への遷移時は復帰先URLを保持しない。
- 復帰先URLはログイン成功直後に1回だけ利用し、利用後に即時破棄する。
- 復帰先URLが不正（外部URL・不許可パス・期限切れ）の場合は既定画面へフォールバックする。

## ログイン/ログアウトの最小フロー
- ログイン
  - フォーム入力を送信し、`laravel(backend)` で認証する。
  - 成功時は認証状態確認APIを再取得し、roleに応じた初期画面へ遷移する。
- ログアウト
  - ログアウトAPIを呼び出し、セッション破棄後に public ルートへ遷移する。

## 変更時チェック
- role定義変更時に `04_role.md` と本仕様の整合を確認する。
- ルートガード変更時に `03_react.md` のURL/ルーティング方針との整合を確認する。
- `401/403/404` 遷移先が `view/errors` 実装と一致することを確認する。
- 復帰先URLの保持・利用・破棄の各条件が実装と一致することを確認する。
